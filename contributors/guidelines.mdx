## Task Creation Guidelines

### Task Design Principles

<Steps>
  <Step title="Real-World Relevance">
    Tasks should represent real-world terminal operations that AI agents might encounter in practice.
  </Step>
  <Step title="Clear Objectives">
    Each task must have well-defined success criteria and expected outputs.
  </Step>
  <Step title="Appropriate Difficulty">
    Tasks should be challenging but achievable, with clear difficulty progression.
  </Step>
  <Step title="Reproducible">
    Tasks must be reproducible across different environments and systems.
  </Step>
  <Step title="Automated Evaluation">
    Tasks should include automated verification methods where possible.
  </Step>
</Steps>

## Quality Standards

### Task Quality Checklist

- [ ] **Clear Instructions**: Step-by-step instructions that are easy to follow
- [ ] **Prerequisites Listed**: All required tools and knowledge clearly stated
- [ ] **Success Criteria**: Well-defined criteria for task completion
- [ ] **Expected Output**: Clear description of what success looks like
- [ ] **Troubleshooting**: Common issues and solutions provided
- [ ] **Testing**: Task has been tested and verified to work
- [ ] **Documentation**: Proper formatting and comprehensive documentation

### Code Quality Standards

- [ ] **Clean Code**: Well-structured, readable, and maintainable code
- [ ] **Comments**: Adequate comments explaining complex logic
- [ ] **Error Handling**: Proper error handling and validation
- [ ] **Testing**: Unit tests where appropriate
- [ ] **Documentation**: Inline documentation and README files

## Submission Process

<Steps>
  <Step title="Fork and Clone">
    Fork the MCPBench repository and clone it to your local machine.
  </Step>
  <Step title="Create Branch">
    Create a new branch for your contribution: `git checkout -b feature/your-contribution`.
  </Step>
  <Step title="Make Changes">
    Implement your changes following our guidelines and quality standards.
  </Step>
  <Step title="Test Thoroughly">
    Test your changes to ensure they work correctly and don't break existing functionality.
  </Step>
  <Step title="Submit Pull Request">
    Create a pull request with a clear description of your changes.
  </Step>
  <Step title="Address Feedback">
    Respond to code reviews and make requested changes.
  </Step>
</Steps>

## Pull Request Guidelines

### PR Description Template

```markdown
## Description
Brief description of the changes made.

## Type of Change
- [ ] New task
- [ ] Bug fix
- [ ] Documentation update
- [ ] Infrastructure improvement
- [ ] Other (please describe)

## Testing
- [ ] Task has been tested locally
- [ ] All existing tests pass
- [ ] New tests added if applicable

## Checklist
- [ ] Code follows project style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] No breaking changes (or clearly documented)
```

### Review Process

1. **Automated Checks**: All PRs must pass automated tests and linting
2. **Code Review**: At least one maintainer must approve the changes
3. **Testing**: Changes must be tested in the evaluation environment
4. **Documentation**: All changes must be properly documented

## Community Guidelines

### Code of Conduct

- **Be Respectful**: Treat all community members with respect and kindness
- **Be Constructive**: Provide helpful feedback and suggestions
- **Be Collaborative**: Work together to improve the project
- **Be Professional**: Maintain a professional tone in all interactions

### Communication

- **Discord**: Use Discord for real-time discussions and quick questions
- **GitHub Issues**: Use GitHub issues for bug reports and feature requests
- **GitHub Discussions**: Use GitHub discussions for general questions and ideas
- **Email**: Use email for sensitive or private matters
